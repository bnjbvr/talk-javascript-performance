JavaScript et performances

asm.js et le futur de JavaScript

Benjamin Bouvier, Mozilla Corporation, Paris, 2014-06-18

Il était une fois JavaScript

  • Langage inventé par /be en 10 jours
  • Implémenté dans tous les navigateurs actuels
  • Nerf de la guerre de la performance
  • La rencontre entre JavaScript et la compilation à la volée

Des applications web complexes

  • Clients mails (roundcube, gmail)
  • Réseaux sociaux (facebook, diaspora, twitter)
  • Cartes géographiques annotées (openstreetmaps, gmaps)
  • L'idée : transformer des applications natives en applications web

Pourquoi JS ?

  • Brique de base du développement web : implémenté et optimisé dans tous les navigateurs.
  • Multi-paradigmes, haut-niveau : ramasse-miettes, typage dynamique, à prototype (~objet).
  • Les interpréteurs et compilateurs se soucient de la performance à votre place.
  • Portable : prêt pour x86, x64, ARM et donc les téléphones mobiles.
  • Mises à jour simplifiées, distribution aisée.

JavaScript comme concurrent du natif ?

  • Performances moins prédictibles que le natif (GC).
  • Le caractère dynamique du langage empêche des optimisations.
  • Caractère très gourmand des applications natives.

asm.js

  • Un sous-ensemble de JavaScript, pas un nouveau langage
  • Typé statiquement
  • Très bas niveau
  • Objectif : assurer des hautes performances de manière prédictible, dans tous les navigateurs

emscripten

moar emscripten

  • Basé sur LLVM
  • Compile du C / C++ vers du JavaScript (portage)
  • Peut générer de l'asm.js
  • Le portage nécessite peu de modifications du code initial
  • Associe des APIs HTML5 aux appels systèmes et graphiques
  • Facilite le portage d'applications existantes pour le web
